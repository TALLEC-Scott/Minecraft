Flow direction

Water has three flow states beyond its level:

  • Source (level 0, no falling flag) — static, no direction. The block is immovable and fuels everything downstream.
  • Falling (WATER_FALLING_FLAG set) — implicit direction is straight down. Treated as effective level 0 when it lands, so it fans out like a fresh source.
  • Flowing (level 1–7) — horizontal direction derived from the level gradient. For each cardinal neighbor, the flow vector accumulates a contribution proportional to how much further from the source that neighbor is. Air counts as maximum drop (a cliff edge). The resulting vector points where the water is going.

The real game engine uses the flow vector for entity physics (players/mobs get pushed in the flow direction) and to tilt the top face of flowing water so it visibly slopes downhill. The 3D demo above visualizes it with the yellow arrows; the render pipeline in src/chunk.cpp computes a per-vertex height from waterFlowLevel but does not yet tilt the top face based on flow direction — a worthwhile follow-up.

How it works

Water in this engine uses an 8-level cellular automaton matching Minecraft's classic model. Each block has a waterLevel field (stored per-chunk in a uint8_t[]). Level 0 is a source: it spreads forever. Levels 1–7 are flowing: each step away from a source increments the level, and level 8 doesn't exist — the block turns to air.

A WaterSimulator instance holds two unordered_set<ivec3>s (current & next-active). The simulator ticks every 15 frames (4 Hz, matching Minecraft's redstone tick), up to 512 blocks per tick, with overflow carried to the next tick so a large flood can't stall the render loop. Neighbor activation runs on block break to wake water into newly-opened space.

The three rules

1 · Gravity

If the block below is air, water falls into it. The falling column keeps the source above alive as long as the chain is unbroken.

2 · Horizontal spread

If current level is < 7, spread to each of the four horizontal neighbors at level + 1, overwriting higher-level water but never lower-level.

3 · Decay

A non-source block with no water above it and no lower-level horizontal neighbor has no support — it increments its own level each tick. At level > 7 it becomes air.

Rendering

Water renders in a second pass after opaque geometry, with blending enabled and face-culling disabled so you can see both sides of the surface. Block height is computed per-vertex as (8 − level) / 9, giving the characteristic sloped-top look for flowing water. The water mesh is part of the same chunk VBO as opaque blocks but uses a separate index range (waterIndexOffset/waterIndexCount) so the renderer can draw it in a second pass without another upload.

Review findings

A pre-release audit of src/WaterSimulator.cpp and tests/test_water.cpp surfaced several issues. The showcase video above uses a corrected 2D replay so the intended behavior is visible — the real simulator still has the bugs below.

  • Fixed Falling water becomes a source. In WaterSimulator.cpp:68, world->setBlock(x, y-1, z, WATER, 0) writes level 0 into the block below. Fixed by introducing WATER_FALLING_FLAG = 0x80 in include/cube.h: the water level byte now carries both a flow level (bits 0-2) and a "falling" flag (bit 7). Rule 1 writes WATER_FALLING_FLAG into the block below instead of 0, so that cell is not a source. Rule 3 was reworked to handle three distinct states — true source (never decays), falling (disappears the moment its chain above breaks), and flowing (normal decay). Chunk mesh building masks the flag when reading level for height computation; falling water renders as full-height.
  • Fixed Period-2 oscillation near landed falling water. Surfaced by running the 3D demo and noticing the active counter never reached zero — a single corner kept flipping between two states diagonally. A standalone Node.js replica (scripts/water_ca_debug.mjs) reproduced it and detected a period-2 cycle at tick 21: flowing L1 cells next to a landed falling column were decaying to L2 each tick, then the falling cell's Rule 2 spread (effective level 0) wrote them back to L1 the next tick. Cause: Rule 3's support scan had if (isFalling) continue;, excluding falling neighbors from counting as support. But since a landed falling cell spreads horizontally at level 0, it is effectively a source for its neighbors. Fixed by treating any falling neighbor as support — the sim now converges in 19 ticks and stays static.
  • Fixed Flowing water sheets over cliff edges. Initial version of the fix for the above introduced a landed check that only applied to falling cells. Flowing cells above empty space happily kept spreading sideways even though the block below was air, producing an umbrella of water hanging in mid-air off each pedestal edge. Reworked so landed applies to every water cell: a cell can only spread horizontally if the block below is solid, the world floor, or a non-falling water cell (pool surface). Air or falling-water below means "mid-column / over a cliff, don't fan out."
  • Critical The tests don't test the real code. tests/test_water.cpp defines a parallel WaterWorld struct that reimplements the CA from scratch — and omits Rule 3 (decay) entirely. All 8 tests pass against this shadow implementation, so bugs in WaterSimulator.cpp are invisible to CI. The tests should construct a real World/ChunkManager/WaterSimulator and exercise WaterSimulator::tick() directly.
  • Fixed "Water above = support" is too permissive. Rule 3 used to treat any WATER block above as structural support regardless of that block's own level, so a random flowing cell above could prop up a cell below even though it had no path to the source. Split into two helpers: hasAnyWaterAbove() for falling cells (any water keeps the chain alive, and the topmost cell's decay cascades down naturally) and hasUpstreamAbove(level) for flowing cells (only counts as support if above is a source, a falling cell, or a flowing cell with level strictly less than ours — i.e., genuinely upstream).
  • Fixed Manual water placement didn't wake the CA. World::destroyBlock called waterSimulator->activateNeighbors() but the symmetric World::placeBlock path did not. Placing a water bucket on the ground could leave the new source sitting inert until something adjacent changed. Fixed: placeBlock now calls activate() on the placed cell when type == WATER and activateNeighbors() for any block placement (so adjacent water also reacts when a solid block blocks its flow). Chunk::placeBlock also now resets waterLevel at the slot so stale data from previous water doesn't bleed through.
  • Fixed Duplicated chunk-coord computation. Rule 1 and Rule 3's "look at block above" paths recomputed worldToChunk(x)/worldToChunk(z) into local byCX/acx variables that duplicated the outer chunkX/chunkZ. Cleaned up during the falling-flag refactor — Rule 1 and the landed check now use the outer chunk pointer and lx/lz directly. Only the neighbor loops in Rule 2 / Rule 3 still recompute, because neighbors can legitimately cross chunk boundaries.
  • Fixed IVec3Hash collides frequently near the origin. x ^ (y << 10) ^ (z << 20) gave many collisions for small coordinates because the shifted bits were mostly zero. Replaced with a boost-style hash_combine: seed ^= h(v.y) + 0x9e3779b9 + (seed << 6) + (seed >> 2), applied to each component.

Reproducing the video

# Build the 2D visualizer (standalone, no OpenGL)
g++ -O2 -std=c++20 scripts/water_demo.cpp -o build/water_demo

# Run: dumps 106 binary PPM frames
mkdir -p docs/water_demo/frames
./build/water_demo docs/water_demo/frames

# Stitch into MP4 with ffmpeg
ffmpeg -y -framerate 15 \
  -i docs/water_demo/frames/frame_%04d.ppm \
  -c:v libx264 -pix_fmt yuv420p -movflags +faststart \
  docs/water_demo/water_physics.mp4

The 2D demo deliberately deviates from the real simulator in two places, both commented in scripts/water_demo.cpp:

  1. Rule 3 decay gate uses !isSource instead of level > 0, so falling water can decay correctly.
  2. Rule 2 horizontal spread is gated on having solid ground below, so a falling column stays 1 block wide until it lands.

Both of these corrections are what the real simulator should be doing.
